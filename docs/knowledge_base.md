# Локальная база знаний AI-помощника

Локальный помощник использует RAG-подход: он ищет релевантный контекст в
проверенных документах проекта и передает этот контекст локальному provider.
Это безопаснее, чем сразу обучать модель, потому что методики можно обновлять,
проверять тестами и просматривать в git.

## Manifest источников

Файл:

```text
config/knowledge_sources.json
```

В нем для каждого источника указаны:

- `path` - относительный путь к markdown-документу;
- `title` - человекочитаемое название;
- `status` - `approved`, `draft` или `reference`;
- `priority` - вес источника при поиске;
- `topics` - ключевые темы для поиска;
- `description` - зачем источник нужен помощнику.

Пути должны быть относительными и не должны выходить за пределы проекта.
Preflight проверяет, что все файлы существуют.

## Проверить базу знаний

```powershell
python scripts/knowledge_base.py
python scripts/knowledge_base.py --query "Как считается Wh?"
python scripts/knowledge_base.py --json
```

Общая проверка проекта:

```powershell
python scripts/preflight.py
```

## Как добавлять знания

1. Добавить или обновить markdown-документ в `docs/`.
2. Добавить источник в `config/knowledge_sources.json`.
3. Указать темы и приоритет.
4. Запустить тесты:

```powershell
python -m pytest tests/test_knowledge_manifest.py tests/test_ai_assistant.py
python scripts/preflight.py
```

5. Обновить `CHANGELOG.md`, если знание влияет на пользователя.

## Что нельзя добавлять

- сырые пользовательские таблицы;
- приватные данные скважин;
- пароли, токены, ключи;
- неподтвержденные формулы без пометки `draft`;
- окончательные геологические заключения без экспертной проверки.

## Дальнейшее развитие

Следующий безопасный шаг - добавить экспертно проверенные Q/A-примеры и
использовать их как отдельный источник RAG. Fine-tune или LoRA стоит делать
только после накопления очищенного и проверенного датасета.
