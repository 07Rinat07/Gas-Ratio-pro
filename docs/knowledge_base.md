# Локальная база знаний AI-помощника

Локальный помощник использует RAG-подход: он ищет релевантный контекст в
проверенных документах проекта и Q/A-примерах, а затем передает этот контекст
локальному provider. Это безопаснее, чем сразу обучать модель, потому что
методики можно обновлять, проверять тестами и просматривать в git.

## Manifest источников

Файл:

```text
config/knowledge_sources.json
```

В нем для каждого markdown-источника указаны:

- `path` - относительный путь к документу;
- `title` - человекочитаемое название;
- `status` - `approved`, `draft` или `reference`;
- `priority` - вес источника при поиске;
- `topics` - ключевые темы для поиска;
- `description` - зачем источник нужен помощнику.

Пути должны быть относительными и не должны выходить за пределы проекта.
Preflight проверяет, что все файлы существуют.

## Q/A-примеры

Файл:

```text
config/knowledge_qa.json
```

Q/A-примеры нужны для типовых вопросов, где пользователю нужен короткий,
проверяемый ответ: почему появился `NaN`, как исправить mapping, что значит
предупреждение по `Ch`, как подготовить Ollama без интернета.

Каждый пример содержит:

- `id` - стабильный идентификатор;
- `question` - типовой вопрос пользователя;
- `answer` - проверяемая инженерная подсказка;
- `status` - статус примера;
- `priority` - вес при поиске;
- `topics` - темы для поиска;
- `sources` - документы, на которых основан ответ;
- `safety_notes` - ограничения и предупреждения.

Q/A-примеры становятся отдельными chunks в поиске и могут попадать в prompt
локального помощника вместе с markdown-документацией.

## Проверить базу знаний

```powershell
python scripts/knowledge_base.py
python scripts/knowledge_base.py --query "Как считается Wh?"
python scripts/knowledge_base.py --query "Почему Wh стал NaN из-за C2?"
python scripts/knowledge_base.py --json
python scripts/evaluate_ai.py
```

Общая проверка проекта:

```powershell
python scripts/preflight.py
```

## Как добавлять знания

1. Добавить или обновить markdown-документ в `docs/`.
2. Добавить источник в `config/knowledge_sources.json`.
3. Для типового пользовательского вопроса добавить пример в `config/knowledge_qa.json`.
4. Указать темы, приоритет, источники и safety notes.
5. Запустить тесты:

```powershell
python -m pytest tests/test_knowledge_manifest.py tests/test_knowledge_qa.py tests/test_ai_assistant.py tests/test_ai_evaluation.py
python scripts/evaluate_ai.py
python scripts/preflight.py
```

6. Обновить `CHANGELOG.md`, если знание влияет на пользователя.

## Что нельзя добавлять

- сырые пользовательские таблицы;
- приватные данные скважин;
- пароли, токены, ключи;
- неподтвержденные формулы без пометки `draft`;
- окончательные геологические заключения без экспертной проверки.

## Дальнейшее развитие

Следующий безопасный шаг - расширять Q/A-примеры после проверки инженером и
измерять качество ответов на выбранной Ollama-модели. Fine-tune или LoRA стоит
делать только после накопления очищенного и проверенного датасета.
