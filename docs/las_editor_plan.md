# План LAS-редактора

Цель: сделать отдельную вкладку `LAS-редактор`, где пользователь сможет
проверить, исправить и подготовить LAS-файл до расчета коэффициентов и графиков.
Оригинальный файл не должен перезаписываться без явного действия пользователя.

## Статус

Сейчас реализованы первые три технических шага:

- приложение читает LAS-файлы через секции `~Curve`/`~C` и `~ASCII`/`~A`,
  распознает `NULL` и передает кривые в общий pipeline mapping/calculation;
- добавлено core-ядро редактора: проверка глубины, построение целевой сетки,
  добавление недостающих строк и стратегии заполнения `empty`, `top`, `bottom`,
  `average`, `linear`;
- добавлена вкладка `LAS-редактор` в Streamlit: загрузка LAS, выбор шага,
  диагностика, ручная правка, экспорт CSV и сохранение подготовленных данных в
  текущую сессию для расчетного workflow;
- редактор исправляет LAS с убывающей глубиной: строки сортируются по
  возрастанию глубины перед расчетами, сохранением и выгрузкой;
- добавлен отчет по шагу глубины: количество шагов, минимальный, максимальный и
  наиболее частый шаг, а также таблица выбросов между соседними глубинами;
- добавлены массовые операции перед построением сетки: удаление дублей глубин,
  обрезка интервала, замена LAS NULL на пустые значения, сортировка глубины и
  проверка монотонности с журналом выполненных действий;
- добавлен предпросмотр до/после ручной правки и сохраняемый журнал правок для
  локальных версий скважин и подготовленных LAS-версий активного проекта;
- добавлены проверяемые подсказки LAS-редактора по шагу глубины, пропускам,
  NULL-значениям, ручному заполнению, сохранению скважин и выгрузке данных.

Постоянное хранение скважин добавлено: версии сохраняются локально в
`data/wells/`, исходные рабочие данные не коммитятся в git, а пользователь может
скачать сохраненную версию в `CSV`, `XLSX` или `LAS`.

Следующий шаг: связать подсказки LAS-редактора с карточками сохраненных версий, чтобы при открытии версии проекта пользователь видел не только журнал правок, но и перечень обязательных проверок.

## Интерпретационные графики, multi-LAS и печать

Новый источник `docs/mud_gas_analysis_literature.md` расширяет план приложения:

1. Загружать несколько LAS-файлов в рабочий workflow и давать выбор набора данных по имени файла.
2. Хранить подготовленные расчеты в session state, чтобы отдельная вкладка графиков могла работать без повторной загрузки.
3. Строить depth tracks по возрастанию глубины сверху-вниз.
4. Давать быстрый режим автомасштаба и ручной масштаб по глубине и X-оси графика.
5. Показывать интерпретационные маркеры по глубине вместе с таблицей расчетов.
6. Экспортировать графики в HTML, который можно открыть в браузере и распечатать для всего интервала или выбранного диапазона.
7. Следующим этапом расширить mapping под GR, total gas, resistivity shallow/deep, density/neutron и другие ГИС-кривые из LAS.
8. Для полноценной корреляции нескольких LAS добавить общий экран сравнения скважин, синхронный диапазон глубин и выбор сохраненных версий.

Подробный план вынесен в `docs/las_correlation_plan.md`.

## Что должен уметь редактор

1. Загружать LAS и показывать кривые, единицы измерения, диапазон глубин и
   текущий шаг.
2. Проверять глубину: дубликаты, обратный порядок, пропуски, неравномерный шаг,
   пустые строки и значения `NULL`. Отчет по шагу показывает минимальный,
   максимальный и наиболее частый шаг, а также интервалы-выбросы для ручной
   проверки. Массовые операции позволяют безопасно удалить дубли, обрезать
   интервал, заменить LAS NULL на пустые значения, отсортировать глубину и
   проверить монотонность.
3. Менять шаг глубины: например `0.1 -> 0.2`, `1.2 -> 2.0`, `2.0 -> 1.0` и
   любой пользовательский шаг в метрах.
4. Автоматически добавлять недостающие строки по выбранной сетке глубин.
5. Заполнять добавленные строки по выбранному правилу:
   `оставить пусто`, `значение сверху`, `значение снизу`, `среднее сверху/снизу`,
   `линейная интерполяция` для числовых кривых.
6. Давать ручной режим: пользователь видит добавленные строки и может изменить
   значения перед расчетами и графиками.
7. Экспортировать исправленный LAS и расчетный CSV, сохраняя журнал правок.

## База скважин и выгрузка

В проект нужно добавить отдельный слой хранения скважин и данных, чтобы работа
не терялась после закрытия приложения.

Планируемые возможности:

1. Создать карточку скважины: название, куст/площадь, дата, источник данных,
   комментарий и статус проверки.
2. Сохранять для скважины исходный LAS, исправленный LAS, расчетную таблицу,
   mapping колонок, журнал правок и предупреждения.
3. Показывать список сохраненных скважин с поиском и фильтрами.
4. Давать выбор одной или нескольких скважин для дальнейшей работы.
5. Выгружать выбранные данные в форматы `LAS`, `XLSX`, `CSV` и позже в отчеты
   `PDF/PNG/SVG`.
6. Разделять исходные данные и исправленные версии, чтобы можно было вернуться
   к оригиналу и понять, какие правки были внесены.
7. Сохранять версию методики/конфига, по которой сделан расчет, чтобы старые
   выгрузки можно было воспроизвести.

Минимальная реализация для начала: локальное хранение в папке `projects/` или
`data/wells/` с manifest-файлом скважины. Полноценную базу данных стоит вводить
после уточнения пользовательского workflow и объема данных.

## Универсальность шагов

Редактор не должен быть привязан только к `0.1` или `0.2` метра. Целевой шаг
выбирается из быстрых вариантов и из ручного поля:

```text
0.1, 0.2, 0.5, 1.0, 1.2, 2.0, пользовательский шаг
```

Числа с запятой (`1,2`) и с точкой (`1.2`) должны приводиться к одному
внутреннему числовому формату.

## Безопасные правила обработки

- Исходный LAS сохраняется как неизмененный источник.
- Все операции редактора фиксируются в журнале: шаг, диапазон, добавленные
  глубины, способ заполнения.
- Для газовых компонентов по умолчанию нельзя молча придумывать значения:
  автоматическое заполнение должно быть явно выбрано пользователем.
- Если данных сверху или снизу нет, редактор оставляет ячейку пустой и показывает
  предупреждение.
- Перед передачей в расчет приложение показывает сводку: сколько строк добавлено,
  сколько значений заполнено автоматически и сколько осталось пустыми.

### Этап 1. LAS import

### Этап 2. Core LAS editor

- добавить модуль `las_editor`;
- реализовать проверку глубины;
- реализовать построение целевой сетки глубин;
- реализовать добавление строк;
- реализовать стратегии заполнения;
- покрыть функции unit-тестами.

### Этап 3. Streamlit-вкладка

- добавить вкладку `LAS-редактор`;
- показать диагностику глубин;
- добавить выбор целевого шага;
- добавить ручную таблицу правок;
- передавать исправленный набор данных в расчетный workflow.

### Этап 4. Хранение скважин

- добавить структуру локального проекта;
- сохранять карточку скважины и версии данных;
- добавить список скважин и выбор сохраненной скважины в интерфейсе;
- покрыть сохранение и загрузку тестами.

Статус: базовая реализация готова через `data/wells/`, manifest скважины,
версии данных и скачивание `CSV`/`XLSX`/`LAS`.

### Этап 5. Экспорт

- экспортировать исправленный LAS;
- экспортировать `XLSX` и `CSV` по выбранным скважинам;
- экспортировать журнал правок для диагностики;
- добавлять строки только в выбранном интервале с отдельной записью в журнале правок;
- подготовить основу для PDF/PNG/SVG отчетов.

### Этап 6. Проверяемые подсказки

Статус: базовая реализация готова. В интерфейсе есть раскрывающийся блок
`Проверяемые подсказки LAS-редактора`. Он показывает статус `ok`, `info`,
`review` или `warning` для следующих проверок:

- шаг глубины: согласованность частого, минимального и максимального шага;
- пропуски глубины: интервалы, где выбранный шаг требует ручной проверки;
- `NULL`: замена LAS NULL и пустые/нечисловые глубины;
- ручное заполнение: добавленные строки выбранного интервала и стратегия заполнения;
- предпросмотр правок: количество добавленных, удаленных и измененных ячеек;
- сохранение скважины: напоминание про название, версию, комментарий и журнал;
- выгрузка данных: контроль exported `CSV/LAS/XLSX` во внешнем просмотрщике.

Каждая подсказка реализована как правило в `las_editor/depth_grid.py` и покрыта
unit-тестами в `tests/test_las_editor.py`.

## Curve Manager: переименование кривых

В LAS-редактор добавлен первый блок профессионального Curve Manager — `Rename curves`. Пользователь выбирает существующую кривую, вводит новое имя и применяет rename до дальнейшей подготовки глубины, массовых операций и ручной правки. Новое имя нормализуется: пробелы по краям удаляются, внутренние группы пробелов заменяются символом `_`, а регистр сохраняется.

Правила безопасности:

- исходная кривая должна существовать в текущем LAS DataFrame;
- новое имя не может быть пустым;
- новое имя не должно совпадать с другой существующей кривой;
- исходное имя сохраняется в истории rename;
- для последнего rename доступен одноуровневый undo, если исходное имя не занято.

История хранит `old_name`, `new_name`, `timestamp`, `reason` и `source`. При rename обновляются ссылки в реально используемых структурах текущего редакторского workflow: tablet tracks, preset-представления, export columns и manifest текущих кривых. Базовая функция обновления ссылок рекурсивно поддерживает dict/list/tuple/set и подготовлена для дальнейших templates, saved calculations и export metadata.
