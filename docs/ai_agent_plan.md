# План локального ИИ-помощника

Цель: добавить локального бесплатного ИИ-помощника, который помогает пользователю
понимать расчеты, ограничения методики, формат данных, предупреждения приложения
и предварительную интерпретацию. Помощник не должен выдавать окончательное
геологическое заключение.

## Статус реализации

Сейчас реализован offline-first skeleton:

- `offline-docs` provider работает без интернета и без модели;
- локальная knowledge base ищет ответы по manifest `config/knowledge_sources.json`;
- CLI `python scripts/knowledge_base.py` показывает источники, Q/A-примеры и результаты поиска;
- `config/knowledge_qa.json` содержит первые проверяемые Q/A-примеры для типовых вопросов;
- `ollama` provider готов для локального `localhost` runtime;
- локальная модель `qwen3:4b` проверена через Ollama на configured evaluation;
- Streamlit показывает блок `Локальный ИИ-помощник`;
- полная таблица и сырые строки файла не передаются помощнику;
- тесты покрывают settings, provider contract, disclaimer и безопасный контекст интервала.

Следующий шаг: расширять curated Q/A-примеры по реальным вопросам пользователей
и постепенно переводить ответы помощника к структурированному формату: кратко,
ограничения, следующие проверки.

## Важный принцип

Сначала делаем помощника как RAG/knowledge assistant по документации проекта,
формулам и подтвержденным методикам. Fine-tune или LoRA имеет смысл только
после накопления проверенного датасета вопросов/ответов и экспертной разметки.

## Возможные локальные runtime

### Вариант A: Ollama

Почему рассмотреть первым:

- простой локальный запуск моделей;
- есть локальный REST API;
- стандартный локальный endpoint обычно работает на `http://localhost:11434`;
- есть Python-клиент;
- удобно подключать к Streamlit.

Официальные ссылки:

- https://github.com/ollama/ollama
- https://github.com/ollama/ollama/blob/main/docs/api.md
- https://ollama.com/library

### Вариант B: llama.cpp

Почему держать как fallback:

- низкоуровневый и гибкий runtime;
- работает с локальными GGUF-моделями;
- есть `llama-server` с HTTP API;
- полезен, если нужно жестко контролировать упаковку и запуск.

Официальные ссылки:

- https://github.com/ggml-org/llama.cpp

## Что помощник должен уметь

1. Объяснять формулы `Wh`, `Bh`, `BAR2`, Pixler ratios, ternary ratios.
2. Объяснять предупреждения приложения.
3. Помогать пользователю подготовить CSV/XLSX.
4. Объяснять, почему некоторые значения стали `NaN`.
5. Давать предварительные инженерные подсказки по выбранному интервалу.
6. Всегда указывать, что интерпретация требует проверки по ГИС, литологии,
   буровому контексту, фону, СПО, наращиваниям и рециркуляции.
7. Ссылаться на локальные документы проекта: `docs/formulas.md`,
   `docs/data_format.md`, `docs/user_guide.md`.

## Что помощник не должен делать

- придумывать новые геологические формулы;
- скрывать неопределенность;
- выдавать окончательное заключение о продуктивности пласта;
- отправлять данные во внешние облачные API без явного разрешения;
- логировать сырые данные скважин и полные пользовательские таблицы.

## Этапы внедрения

### Этап 1. Offline knowledge assistant

- добавить интерфейс `LocalAssistant`;
- собрать контекст из локальных docs;
- сделать fake provider для тестов;
- добавить Streamlit-блок "Помощник";
- отвечать только на вопросы по документации проекта.

### Этап 2. Ollama integration

- добавить настройку `AI_PROVIDER=ollama`;
- добавить `OLLAMA_BASE_URL=http://localhost:11434`;
- добавить проверку доступности локального сервера;
- сделать понятную ошибку, если Ollama не установлен или модель не загружена;
- покрыть клиент unit-тестами через mock HTTP.

### Этап 3. Context-aware interval assistant

- передавать помощнику только выбранный интервал и расчетные коэффициенты;
- не передавать всю таблицу без необходимости;
- добавлять источники: формулы, ограничения, предупреждения;
- требовать structured response: summary, caveats, next checks.

### Этап 4. Обучение на проектных данных

Начать не с fine-tune, а с curated knowledge base:

- подтвержденные методики;
- примеры корректной интерпретации;
- типовые ошибки импорта;
- экспертно проверенные Q/A.

Fine-tune рассматривать только после:

- согласования лицензий и приватности данных;
- очистки чувствительных данных;
- экспертной проверки датасета;
- появления метрик качества.

## Минимальные тесты для AI-фич

- помощник не отвечает без disclaimer;
- prompt содержит ссылку на ограничения интерпретации;
- provider interface можно заменить fake provider;
- при недоступном Ollama UI показывает понятную ошибку;
- в лог не попадают полные таблицы и сырые строки файла;
- ответы по формулам основаны на `docs/formulas.md`.

## Решение на сейчас

В ближайшей версии не скачивать модель автоматически. Сначала подготовить
архитектуру, тесты, prompt contract и RAG по локальным документам. После этого
выбрать конкретную модель под железо пользователя и добавить инструкцию установки.
